四種協作場景 · 取自實戰對話

把實際與 AI 的對話濃縮成精華 — 看得出每次的角色分工。

SCENE 01 · 規格比對 (BGD AP→GL)
附上越南版 AP→GL 規格、BGD 這版規格 — 幫我比對差異點,產出 markdown。
已完成比對。三個關鍵差異:
  • 帳本 ID:803 → 804 (BGD)
  • P_USER:VINUSER → RITAWANG
  • GL date:Hardcode → 參數化區間
其他流程與越南版一致。MD 已產出,要我貼出來嗎?
好,沿越南 VIN 的命名風格產出 procedure。
SCENE 02 · Coder → Reviewer
Code 你寫,我 Review + 對答案。
已產出完整 PKG,帳本 / USER / 日期皆參數化:
APPS.MK_AP_RPA_IMPORT_GL_PKG.MAIN_BGD ( ERRBUF => V_ERRBUF, RETCODE => V_RETCODE, P_SOB_ID => 804, -- BGD 帳本 P_USER => 'RITAWANG', -- BGD User P_DATE_FM => '2026/04/01', P_DATE_TO => '2026/04/27' );
✓ Review 通過。執行範例傳給我,我在 Test 區跑一輪。
預期傳票筆數 / 金額已對好,結果若一致就 cc 王凰凰結案。
→ 32 HR (預期) → 3 HR (實際)
SCENE 03 · AI 規劃 feature (主副料 IG)
我想做 unAPInvoice 清單頁,要可以條件搜尋、勾選後一鍵 Import 拋帳。
建議用 APEX Interactive Grid + JS 客製。前後端怎麼搭:
// 前端:Vendor / InvNo / Etd 範圍 apex.region("un_invoice_ig").refresh(); // 勾選後送 APEX_PROCESS apex.server.process("DO_IMPORT", {x01: rowIds});
後端 PL/SQL 我也順便 Review 了 schema,master/detail 連動建議用 ROW_ID。
手把手教我怎麼調前端
SCENE 04 · 規格在討論中變完整 (商品存貨)
MTS 模式下,我要記錄商品存貨的進銷存,成本怎麼歸屬?
反問先把規格挖深三個點:
  1. cost layer:月平均、FIFO、還是標準成本?
  2. 成品入庫的時點:dispatch_note 確認時 vs 工單關單時?
  3. 退貨怎麼回推?要不要保留批次?
FIFO、dispatch_note 確認時、批次保留。
那建議新增 inv_cost_layer 表, 結構與物件清單我整理在下面 — 你 Review 完我直接寫 SQL。
→ 規格在來回討論中被想得更完整 = 最大副產品

H1 SUMMARY

四個關鍵收穫

從專案到方法論

01

AI 協作的兩種價值

BGD AP→GL 速度 (32 HR → 3 HR) 與商品存貨 品質 (規格更完整) 兩個案例, 讓我學到 AI 既是「省工時」工具,也是「規格設計」助手。

02

Coder → Reviewer

BGD 拋帳自動化讓我意識到工作正從「敲鍵盤」轉為「對答案」; 未來會把更多時間花在思考、Review、與 USER 對話。

03

規格完整度 比 工時 更值錢

商品存貨改版讓我看到「規格在來回討論中被想得更完整」是最大副產品; 未來把跟 AI 討論規格當成 SA 階段必要步驟。

04

需求先 MD 化,讓 AI 抓盲點

未來與使用者合作,可先將使用者的需求轉化成 MD, 透過 AI 找出邏輯盲點,與使用者溝通後再進行開發。
